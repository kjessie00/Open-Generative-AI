# 앱 안에서 기획·설계 에이전트와 함께 작업하기

## 한눈에 보는 현재 파이프라인

| 단계 | 앱에서 지금 할 수 있는 일 | 현재 실행 경계 |
| --- | --- | --- |
| 1 기획·대본 | 직접 작성·저장, 에이전트에게 수정 요청, 현재 내용과 수정안 비교, 적용 또는 현재 내용 유지 | 로컬 Codex가 앱 버튼에서 실제 실행됨 |
| 2 설계 | 인물 시트 → 장소 시트 → 장면 카드를 직접 편집·저장, 전체 설계 수정 요청, 비교·적용 | 로컬 Codex가 앱 버튼에서 실제 실행됨 |
| 3 생성 준비 | 이미지·영상 프롬프트 직접 편집, 순차 작업 목록 확인, 기존 DST/영상 결과 연결, 검토할 결과와 다시 만들 항목 선택 | 생성 요청은 dry-run/준비만 가능. 유료·외부 생성은 실행하지 않음 |
| 4 클립 선택 | 연결된 영상 검토, 사용할 구간 선택, 결과 저장 | 로컬 결과가 있어야 진행 가능 |
| 5 마무리 | 선택 구간 기반 로컬 최종 편집·보고 상태 확인 | 사람의 결과 품질 승인과 실제 납품은 별도 |

현재 화면의 핵심 흐름은 `직접 수정` 또는 `에이전트 작업 시작` 중 하나를 고르고, 에이전트 수정안이 오면 원문과 비교해 적용하는 것이다. 내부 요청 ID, 해시, 경로, 모델 이름, 영문 상태 코드, 상태 배지는 표시하지 않는다.

## 사용자 흐름

### 기획·대본

1. 기획이나 스크립트를 직접 고치고 `직접 저장`한다.
2. 도움이 필요하면 `어떻게 바꿀까요?`에 짧게 적고 `에이전트 작업 시작`을 누른다.
3. 앱이 현재 내용을 먼저 저장하고 로컬 Codex에게 해당 부분만 수정하게 한다.
4. `현재 내용`과 `에이전트 수정안`을 비교한다.
5. `수정안 적용` 또는 `현재 내용 유지`를 고른다.

기획 수정안의 `proposed_text`에는 완성된 본문만 허용한다. JSON 객체, `current_brief/current_script` 필드, 코드 블록, 원문과 동일한 결과는 앱이 수정안으로 저장하지 않는다.

### 인물·장소·장면 설계

1. 인물 시트, 장소 시트, 장면 카드를 직접 열어 수정할 수 있다.
2. 전체 연속성이나 카메라·행동을 함께 다듬고 싶으면 설계 요청칸에 적고 `에이전트 작업 시작`을 누른다.
3. 현재 설계와 에이전트 전체 수정안을 나란히 본다.
4. 적용하거나 현재 설계를 유지한다.
5. 앱을 종료하고 다시 열어도 적용한 기획·설계와 결정 상태가 복원된다.

## 실제 로컬 에이전트 실행 계약

- renderer는 셸, 실행 경로, 모델, 출력 파일을 정하지 않는다. `window.filmPipeline`의 고정된 IPC만 호출한다.
- Electron main이 절대 경로의 로컬 Codex CLI를 고정 모델 `gpt-5.3-codex-spark`, reasoning effort `xhigh`로 실행한다.
- 승인 `never`, sandbox `read-only`, ephemeral, 별도 `0700` 작업 폴더를 사용한다.
- shell, unified exec, 앱·플러그인, 브라우저, Computer Use, 이미지 생성, 다중 에이전트, 메모리, hooks, 도구 제안, workspace dependencies를 모두 끈다.
- 프롬프트는 stdin으로만 전달하고, 구조화 출력은 임시 `0600` 파일에서 검증한 뒤 삭제한다.
- 모델은 요청 ID를 받지 않는다. Electron main이 신뢰한 요청 ID를 검증된 출력에 붙여 기존 suggestion provider에 발행한다.
- 타임아웃이나 실패 시 앱이 소유한 자식 process group만 종료한다. 다른 Codex·Claude 세션은 건드리지 않는다.
- UI에는 짧은 한국어 성공·재시도 문장만 보이고 stderr, 프롬프트, 경로, 원시 오류는 노출하지 않는다.

## 실제 검증

격리된 실제 Electron 앱에서 합성 영화 기획·대본과 1장면 설계를 사용해 다음을 검증했다.

- 앱 버튼에서 실제 로컬 Codex 기획 수정 1회, 설계 수정 1회 성공
- 기획 수정안은 JSON 포장이 아닌 순수 본문임을 검사하고 적용
- 설계 수정안은 현재 보드와 다른 전체 인물·장소·장면 보드임을 검사하고 적용
- 기획 적용 직후 설계 revision을 다시 읽어 stale 저장 오류를 방지
- 지원되지 않는 구조화 출력 스키마 옵션을 제거하고 provider 필수 문자열 조건과 스키마를 일치시킴
- 앱 완전 종료·재실행 뒤 기획과 설계 모두 `applied` exact 복원
- suggestion 파일 `app_model_called:true`, `produced_by_agent:true`, mode `0600`; draft root `0700`
- 320/768/1024/1440에서 핵심 작업대 가로 넘침 0, 잘린 조작부 0, 최소 조작 높이 44px, badge 0
- renderer 외부 요청 0, console warning/error 0, exception 0, production 생성 0
- 이미지·영상 provider 호출, 이미지·영상 생성, API 키 사용, 업로드 0

## 현재 상태와 다음 범위

기획·대본과 설계의 앱 내부 실제 에이전트 협업은 기술적으로 PASS다. 실제 생성 provider는 아직 실행하지 않는다. 다음 구현 범위는 이미지 프롬프트/영상 프롬프트에서도 같은 `직접 수정 또는 에이전트에게 요청 → 비교 → 적용` 계약을 확장한 뒤, DST·Flow·Grok·Replicate/ByteDance별 안전한 live executor 계약을 별도로 연결하는 것이다. 이미지·영상 생성 성공, 결과 품질, Jessie의 최종 승인도 서로 다른 상태로 유지한다.
