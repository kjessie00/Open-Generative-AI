# 기획·대본 에이전트 수정안 검토

## 결론

기획·대본 단계에서 사용자가 직접 편집하는 흐름과 에이전트에게 요청하는 흐름을 한 화면에 연결했다. 에이전트 수정안이 도착하면 원문과 수정안을 나란히 보고 `수정안 적용` 또는 `보류`를 선택할 수 있으며, 보류한 수정안은 나중에 다시 열어 적용할 수 있다.

## 사용자 흐름

1. 기획이나 스크립트를 직접 작성하고 저장한다.
2. `에이전트에게 요청`에 바꾸고 싶은 내용을 적는다.
3. 앱은 요청 시점의 기획·대본을 비공개 불변 스냅샷으로 보존한다.
4. 로컬 에이전트가 요청 묶음을 읽고 수정안을 발행한다.
5. 사용자는 `수정안 확인`을 눌러 `현재 내용`과 `에이전트 수정안`을 비교한다.
6. `수정안 적용`은 해당 단계만 바꾸고, `보류`는 원문을 그대로 둔다.
7. 직접 편집한 원문이 달라졌다면 오래된 수정안 적용을 막고 새 요청을 남길 수 있게 한다.

대기·도착·보류·적용 상태는 짧은 한국어 문장으로만 표시한다. 요청 ID, 수정안 토큰, 해시, 파일 경로, 모델/provider 이름, 내부 상태 코드는 화면에 노출하지 않는다.

## 실행 경계

- Electron 앱은 모델, API, 셸 명령, API 키를 직접 호출하지 않는다.
- 앱은 renderer → preload → main 경계로 요청 저장, 상태 조회, 적용·보류 결정만 처리한다.
- 외부 로컬 에이전트는 `scripts/planning_agent_handoff.mjs`의 `prepare`로 요청 시점 스냅샷을 받고, `publish`로 한 요청당 하나의 불변 수정안을 발행한다.
- 스냅샷을 먼저 완성하고 요청 파일을 마지막에 발행한다. 디렉터리는 `0700`, 파일은 `0600`이다.
- 적용 시 main이 저장된 수정안과 현재 대상 단계의 해시를 다시 검증한다. 기획 수정안 적용은 독립적인 대본 수정안을 무효화하지 않는다.
- 적용·보류 결정은 불변 영수증으로 남고, 초안 저장 뒤 영수증 발행이 끊겨도 재시도에서 복구된다.
- 앱 안 요청 레코드는 계속 `executed:false`, `model_called:false`다. 실제 수정안 생성은 앱 밖 현재 Codex가 수행했다.

## 실제 Electron 검증

격리된 Electron `userData`에서 기존 moving-battlefield 기획·대본을 직접 저장하고 기획·대본 요청을 각각 만들었다. 현재 Codex가 실제 요청 문맥을 읽고 다음 수정안을 작성했다.

- 기획: `문의 32통, 사장 몫 0원`의 첫 장면 갈등과 45초 안에 얻을 시청자 약속을 앞부분에 압축
- 대본: 첫 3초를 `문의 32통, 사장 몫 0원. 더 싸게 받아야 할까요?`로 정리

두 수정안을 로컬 CLI로 발행한 뒤 실제 UI에서 원문과 제안의 exact 본문을 확인했다. 기획 수정안은 적용하고 대본 수정안은 보류했다.

- 수정안 도착: 기획·대본 2개, `수정안이 도착했습니다` 2개 PASS
- 기획 결정: `applied`, 초안이 제안 본문과 exact match
- 대본 결정: `held`, 기존 대본 exact 유지
- 완전 종료·재실행: 기획 `applied`, 대본 `held` 및 두 본문 exact 복원
