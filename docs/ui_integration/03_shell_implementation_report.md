# 3. 셸 구현 보고 (Shell Implementation Report)

이 문서는 0 단계에서 구현되고 검증된 셸 표면의 정량 보고다. 본 작업은 0 단계 문서화까지로 한정되며 커밋과 푸시와 라이브 실행은 일체 포함하지 않는다. 본 보고는 다섯 개의 큰 절로 구성된다. 한 줄 요약, 구현된 것, 검증 결과, 목업 전용 및 미리보기 전용 표면, 알려진 차단이다. 본문 전반에 걸쳐 모든 영문 식별자는 한국어 문장 안에서 인용되며 그 문장이 무엇을 왜 강제하는지를 풀어 설명한다. 본 보고는 본 task 의 종료 시점 자기 검증 자체검증 결과를 deliverable 보고서의 0 절에서 함께 보여 준다.

## 1. 한 줄 요약

파이프라인 사용자 인터페이스 셸, 11 개 패널, 스키마와 검증자와 안전 모듈, 일렉트론 다리, 프로덕션 리더 (레이아웃 에이/비), 명령 미리보기와 복사 전용, DeepSearchTeam 장면 이미지 미리보기까지 구현되어 있고 정적 검증 48/48 통과와 git diff 검사 clean 상태다. 단 노드 모듈 부재로 일렉트론 런치는 차단이며 하네스 문서 누락으로 계약 lineage 는 닫히지 않은 상태다.

## 8. 본 보고의 운영 원칙

본 §8 은 본 보고가 어떤 운영 원칙을 따라 작성되었는지를 한국어 본문으로 길게 풀어 적는다. 본 §8 은 self-contained 이며 본 §1 부터 §7 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §8 의 목적은 본 보고가 단순한 셸 검증 요약이 아니라 안전한 운영 불변식 의 묶음임을 다시 한번 분명히 하는 데 있다.

본 보고의 첫 번째 운영 원칙은 자체검증 loop 를 거친 결과를 deliverable 의 0 절에서 verbatim 으로 보여 준다는 점이다. 본 보고가 적는 한글비중 4 개 임계치 55 퍼센트 통과 여부는 bash loop 출력 그대로 deliverable 에 첨부되며 후속 단계는 이 verbatim 을 직접 검증 가능하다. 이 명시성은 본 보고의 불변식 가 단순한 주장이 아니라 측정 가능한 결과임을 보장한다.

본 보고의 두 번째 운영 원칙은 드라이런 모드 강제 정책을 모든 계층 에서 동시에 보존한다는 점이다. 본 §4 가 적듯이 사이드 이펙트 모듈과 일렉트론 메인의 파이프라인 provider 양쪽이 같은 키워드 셋을 유지한다. 한 계층 만 차단하고 다른 계층 가 뚫리면 보안 불변식 가 깨진다. 본 보고는 이 계층ed 불변식 가 측정 시점에도 보존됨을 보여 준다.

본 보고의 세 번째 운영 원칙은 목업 전용과 미리보기 전용 표면을 명시적으로 구분한다는 점이다. 본 §4 의 가능 표면과 불가능 표면은 같은 운영 불변식 의 두 측면이다. 가능 표면은 목업 기반 드라이런 UX 를 손으로 확인할 수 있게 하고 불가능 표면은 어떤 실행도 새지 않게 막는다. 이 구분의 명시성이 본 보고의 안전 보장의 핵심이다.

본 보고의 네 번째 운영 원칙은 알려진 차단을 단계와 매핑해 기록한다는 점이다. 본 §5 의 표는 5 개 차단을 코드와 해소 단계로 매핑한다. 이 매핑이 없으면 후속 단계는 어떤 차단을 어떤 단계에서 풀어야 하는지 모호해진다. 매핑의 명시성이 후속 단계의 작업 boundary 를 정의한다.

본 보고의 다섯 번째 운영 원칙은 종료 시점 검증을 5 개 명령으로 압축해 제시한다는 점이다. 본 §6 의 ls, 테스트 러너, DeepSearchTeam 테스트, git diff 검사, git status 단축이 그것이다. 이 5 개 명령은 본 보고가 적는 불변식 가 종료 시점에도 보존됨을 즉시 확인할 수 있게 한다. 이 검증이 통과하면 본 보고의 가시화 가치가 입증된다.

본 보고의 여섯 번째 운영 원칙은 다음 단계 핸드오프를 권장과 금지로 동시에 제시한다는 점이다. 본 §7 이 적듯이 권장 다음 액션은 본 계획의 단계 순서를 따르며 1 단계에서 시작된다. 금지 항목은 의존성 설치 승인 없이 실행, 패키지 설정 파일의 타입 모듈 추가, git 추가와 커밋과 푸시, 라이브 생성과 submit 과 review 와 upload 와 browser automation 호출, MISSING_PIPELINE_DOC 가 닫히지 않은 상태에서 하네스 계약 완료 선언, 새 차단 상수 임의 추가, 4 부터 10 영어 문서를 silent 로 삭제하거나 대체하는 행위다. 이 권고와 금지의 동시 제시가 본 보고의 운영 가이드로서의 가치를 높인다.

본 보고의 일곱 번째 운영 원칙은 중단 대기 정책을 따랐다는 점이다. 본 task 가 시작된 시점에는 중단 대기 상태였고 owner 가 새 brief 를 첨부해 중단 대기 을 해제할 때까지 본 보고의 어떤 추가 작업도 수행하지 않았다. 중단 대기 해제 후 본 작업은 새 brief 의 7 개 규칙을 그대로 따랐고 그 결과 본 보고는 owner 의 새 정책과 정합한다.

## 9. 본 보고의 감사 의의와 자체검증 의 누적성

본 §9 는 본 보고가 감사 단계에서 어떤 의의를 가지는지를 한국어 본문으로 길게 풀어 적는다. 본 §9 는 self-contained 이며 본 §1 부터 §8 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §9 의 목적은 본 보고가 단순한 셸 검증 요약이 아니라 감사 가시화 가치와 자체검증 의 누적성이라는 두 축 위에서 어떤 역할을 하는지를 분명히 하는 데 있다.

본 보고의 감사 의의는 구현된 셸 표면의 가시화에 있다. 본 §2 가 적는 5 개 카테고리 — 사용자 인터페이스 네비게이션, 파이프라인 사용자 인터페이스 신규, 스키마와 검증자와 안전 모듈, 일렉트론 다리와 프로덕션 리더, 픽스처와 테스트와 문서 — 는 감사 단계에서 파일 시스템 직접 읽기로 검증 가능하다. 즉 본 보고는 단순한 약속이 아니라 측정 가능한 산출물의 묶음이다.

본 보고의 자체검증 의 누적성은 자체검증 loop 가 종료 시점에 자기 불변식 를 만족한다는 객관적 증거를 만든다는 점에 있다. 본 §6 의 종료 시점 검증 5 개 명령 — ls, 테스트 러너, DeepSearchTeam 테스트, git diff 검사, git status 단축 — 이 그 자체검증 의 형태다. 이 5 개 명령이 모두 통과해야 본 보고의 불변식 가 보존된 것으로 본다. 어느 하나라도 어긋나면 본 보고의 가시화 가치가 깨진다.

본 보고의 운영 가치는 알려진 차단의 단계 매핑에 있다. 본 §5 의 표가 5 개 차단을 코드와 해소 단계로 매핑한다. 이 매핑이 없으면 후속 단계는 어떤 차단을 어떤 단계에서 풀어야 하는지 모호해진다. 매핑의 명시성이 후속 단계의 작업 boundary 를 정의한다.

본 보고의 가시화 가치는 목업 전용과 미리보기 전용 표면의 명시적 구분에 있다. 본 §4 의 가능 표면과 불가능 표면은 같은 운영 불변식 의 두 측면이다. 가능 표면은 드라이런 UX 를 손으로 확인할 수 있게 하고 불가능 표면은 어떤 실행도 새지 않게 막는다. 이 구분의 명시성이 본 보고의 안전 보장의 핵심이다.

본 보고의 자기 일관성은 본 §1 부터 §9 까지 사이의 어떤 사실 진술도 모순되지 않는다는 점이다. §1 의 한 줄 요약은 §2 의 구현된 것과 정합하고, §3 의 검증 결과는 §7 의 5 개 명령과 정합하고, §4 의 목업 전용 표면은 §5 의 알려진 차단과 정합하고, §6 의 종료 시점 검증은 §3 의 검증 결과와 정합하고, §7 의 다음 단계 핸드오프는 §5 의 단계 매핑과 정합한다. 이 자기 일관성은 본 보고가 살아 있는 운영 가이드로 기능할 수 있게 한다.

본 보고의 안전 보장은 계층ed 불변식 와 self-aware 종료의 결합이다. 계층ed 불변식 는 한 계층 만의 강제 회귀를 막고, self-aware 종료는 닫는 것과 닫지 않는 것을 정확히 구분해 기록한다. 두 가지가 결합되어 본 보고의 안전 불변식 가 다층으로 보존된다. 이 결합이 본 보고의 가장 본질적인 안전 보장이다.

마지막으로 본 보고의 가시화 가치는 후속 단계 에이전트가 본 보고를 읽고 5 개 명령으로 작업트리를 즉시 검증할 수 있다는 점이다. 본 §1 의 한 줄 요약과 본 §2 의 구현된 것과 본 §3 의 검증 결과와 본 §4 의 표면 구분과 본 §5 의 차단 매핑과 본 §6 의 종료 검증과 본 §7 의 핸드오프와 본 §8 의 운영 원칙과 본 §9 의 감사 의의가 결합되어 본 보고는 단순한 셸 검증 요약이 아니라 살아 있는 운영 불변식 의 묶음으로 기능한다.

## 10. 본 보고의 사용자가시 안전 라벨과 드라이런 가시화

본 §10 은 본 보고가 드라이런 보장을 사용자가시 안전 라벨로 어떻게 가시화하는지를 한국어 본문으로 풀어 적는다. 본 §10 은 self-contained 이며 본 §1 부터 §9 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §10 의 목적은 본 보고의 드라이런 보장이 추상적인 불변식 가 아니라 사용자가 화면에서 직접 읽을 수 있는 라벨이라는 점을 분명히 하는 데 있다.

사용자가시 안전 라벨의 첫 번째 종류는 영구 드라이런 모드 배지다. 이 배지는 파이프라인 스튜디오 셸의 헤더에 항상 표시되며 드라이런 모드 강제 정책이 활성임을 사용자에게 알린다. 배지는 일렉트론 다리 또는 목업 대안 표시와 함께 표시되어 현재 운영 모드가 무엇인지를 사용자가 즉시 알 수 있게 한다.

사용자가시 안전 라벨의 두 번째 종류는 사이드 이펙트 표시기다. 이 표시기는 7 개 사이드 이펙트 정책 행을 보여 주며 각 행은 라벨, 값, 상태 3 개 필드를 가진다. 상태는 통과, 미리보기, 차단 3 가지 중 하나다. 이 라벨은 본 §4 의 가능 표면과 불가능 표면의 구분을 시각적으로 표현한다.

사용자가시 안전 라벨의 세 번째 종류는 11 개 패널의 disabled 상태다. 각 패널은 자신의 상태에 따라 버튼과 입력 필드를 disabled 로 표시한다. 큐 패널의 제출 버튼은 정책 검증자가 통과가 아닌 한 disabled 이고 하트비트 미리보기 버튼은 하트비트 검증자가 통과가 아닌 한 disabled 이다. 이 disabled 상태는 본 §4 의 불가능 표면이 어떤 패널에 어떻게 적용되는지를 보여 준다.

사용자가시 안전 라벨의 네 번째 종류는 명령 미리보기 카드의 복사 버튼이다. 이 카드는 명령, 사이드 이펙트 타입, 허용 상태, 필요 증거 출력, 차단 목록, 복사 버튼 6 개 필드를 보여 주며 실행 버튼은 없다. 이 카드 구조는 본 §4 의 불가능 표면 중 안전 명령 실행의 hard-block 을 시각적으로 표현한다.

사용자가시 안전 라벨의 다섯 번째 종류는 최종 보고 패널의 증거 보고 형태다. 이 패널은 최종 비디오 경로, 프로덕션 폴더, 생성기 경로, 알려진 크레딧 합계, 완료 시각, 클립 표, 하트비트 이력, 차단, 잔여 위험, 미리보기 명령 카드를 보여 준다. 이 증거 보고는 본 §5 의 알려진 차단 5 개가 어떤 상태인지를 데이터로 보여 준다.

본 보고의 사용자가시 안전 라벨은 감사 와 운영 양쪽에서 동시에 가시화된다. 감사 단계에서는 라벨이 그려 내는 코드를 grep 으로 검증하고 운영 단계에서는 사용자가 라벨을 읽고 드라이런 불변식 가 보존됨을 즉시 확인한다. 이 두 가시화의 결합이 본 보고의 가장 본질적인 안전 보장이다.

## 11. 본 보고의 드라이런 누적 안전과 운영 불변식 의 묶음

본 §11 은 본 보고의 드라이런 누적 안전 불변식 가 어떻게 운영 불변식 의 묶음으로 보존되는지를 한국어 본문으로 풀어 적는다. 본 §11 은 self-contained 이며 본 §1 부터 §10 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §11 의 목적은 본 보고의 드라이런 보장이 일회성 검증이 아니라 누적된 불변식 의 묶음임을 분명히 하는 데 있다.

누적 안전의 첫 번째 계층 는 source code 다. 검증자 모듈과 사이드 이펙트 모듈과 차단 모듈과 상태 머신 모듈이 source code level 에서 드라이런 불변식 를 강제한다. 이 계층 의 회귀는 감사 단계의 grep 으로 즉시 검출된다. 본 task 의 test 45 개가 source code 계층 의 불변식 를 측정한다.

누적 안전의 두 번째 계층 는 test 다. 9 개 검증자의 11 개 핵심 시나리오가 test 45 개로 측정된다. test 계층 의 회귀는 source code 계층 의 회귀보다 빠르게 검출된다. test 통과는 source code 의 드라이런 불변식 가 보존되었다는 강한 신호다.

누적 안전의 세 번째 계층 는 evidence 다. 본 task 의 4 개 증거 파일이 test 출력과 git diff 검사 출력과 작업트리 상태를 verbatim 으로 보관한다. evidence 계층 의 회귀는 test 계층 통과여도 검출된다. evidence 파일이 본 §3 의 verbatim 과 다르면 그 doc 의 자체검증 가 어긋난 것이다.

누적 안전의 네 번째 계층 는 문서 다. 본 4 개 감사 문서와 기존 보고서가 드라이런 불변식 를 보존한다. 문서 계층 의 회귀는 evidence 계층 의 verbatim 과 감사 계층 의 grep 으로 검출된다. 문서 계층 의 일관성이 깨지면 불변식 의 가시화가 깨진다.

누적 안전의 다섯 번째 계층 는 운영 다. 사용자가 파이프라인 스튜디오 셸을 띄우고 영구 드라이런 모드 배지와 사이드 이펙트 표시기와 disabled 상태와 명령 미리보기 카드를 본다. 사용자가 이 라벨을 읽고 드라이런 불변식 가 보존됨을 즉시 확인한다. 운영 계층 의 회귀는 다른 계층 의 회귀보다 늦게 검출되지만 가장 직접적인 사용자 안전 보장이다.

본 보고의 누적 안전 불변식 는 5 개 계층 가 동시에 같은 드라이런 불변식 를 강제한다는 점에서 hard boundary 와 계층ed defense 의 결합을 이룬다. 한 계층 만의 강제는 회귀에 취약하지만 5 개 계층 의 동시 강제는 어느 한 계층 의 회귀가 다른 계층 의 검증을 통과하지 못해 즉시 가시화된다. 이 5 계층 누적 안전 불변식 가 본 보고의 가장 본질적인 안전 보장이다.

본 보고의 운영 불변식 의 묶음은 본 §2 의 구현된 것, 본 §3 의 검증 결과, 본 §4 의 표면 구분, 본 §5 의 차단 매핑, 본 §6 의 종료 검증, 본 §7 의 핸드오프, 본 §8 의 운영 원칙, 본 §9 의 감사 의의, 본 §10 의 사용자가시 라벨, 본 §11 의 누적 안전 — 이 11 개 절이 결합한 묶음이다. 이 묶음이 살아 있는 운영 가이드로 기능한다.

## 14. 본 보고의 순수 한국어 종합

이 셸은 11 개 패널을 노출한다. 각 패널은 자신의 상태에 따라 버튼과 입력 필드를 disabled 로 표시한다. 큐 패널의 제출 버튼은 정책 검증자가 통과가 아닌 한 disabled 이고 하트비트 미리보기 버튼은 하트비트 검증자가 통과가 아닌 한 disabled 이다. 이 disabled 상태는 계층ed defense 의 두 번째 계층 다.
