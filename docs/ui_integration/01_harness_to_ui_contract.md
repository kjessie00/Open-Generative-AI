# 1. 하네스 ↔ 사용자 인터페이스 계약 (Harness ↔ UI Contract)

이 문서는 Open-Generative-AI 의 시네마틱 파이프라인 사용자 인터페이스가 외부 하네스 및 원본 파이프라인과 어떤 표면을 교환해야 하는지 정의한 계약이다. 이 문서가 닫히기 전까지는 하네스 계약이 완료되었다고 주장하지 않는다. 본 계약의 핵심 불변식 는 사용자 인터페이스가 자체적으로 영상을 만들지 않는다는 점이다. 사용자 인터페이스는 Jessie 가 별도로 운영하는 하네스의 드라이런 사용자 인터페이스 셸이며 그 하네스의 결과물을 읽고 미리보기 전용 명령 표면을 보여 주는 것이 전부다. 본 계약은 다섯 개의 큰 절로 구성된다. 하네스 측 의무, 사용자 인터페이스 측 의무, 안전 계약, 데이터 흐름, 누락된 하네스 입력 항목이다. 본문 전반에 걸쳐 모든 영문 식별자는 한국어 문장 안에서 인용되며 그 문장이 무엇을 왜 강제하는지를 풀어 설명한다.

## 1. 하네스 측 의무

이 프로젝트의 파이프라인 사용자 인터페이스는 자체적으로 영상을 만들지 않는다. 사용자 인터페이스는 Jessie 가 별도로 운영하는 하네스의 드라이런 사용자 인터페이스 셸이며 그 하네스의 결과물인 파일, 원장, 리뷰, 마크다운 보고를 읽고 미리보기 전용 명령 표면을 보여 주는 것이 전부다. 따라서 사용자 인터페이스가 합리적으로 동작하려면 두 문서가 반드시 로컬에 존재해야 한다. 그 두 문서가 부재하면 본 계약은 닫히지 않는다.

본 계약이 요구하는 두 필수 하네스 문서는 다음과 같다. 첫 번째는 shorts 라는 도메인의 작업 명세로 shorts 계열 프로덕션의 필드 명세와 차단 명세와 안전 규칙 원본을 담는다. 두 번째는 Seedance 2 라는 도메인의 작업 명세로 Seedance 및 Dreamina 계열의 제출과 하트비트와 품질 보정과 수락 초 명세와 게이트 정의 원본을 담는다. 이 두 파일이 없는 상태에서 구현된 스키마와 검증자는 목업 기반 로컬 사용자 인터페이스 스캐폴드이지 하네스 계약이 아니다.

이 두 파일이 부재할 때의 결과는 명시적이다. 사용자 인터페이스가 필드명이나 스키마를 자체 추정해야 하고 Seedance 계열의 제출과 하트비트와 품질 보정과 수락 초 기준이 목업 에서 유래했음을 Jessie 가 검증할 수 없으며 블록 단축 코드 MISSING_PIPELINE_DOC 가 스키마 레이어에 등록되어 활성화된다. 본 프로젝트의 에이전트 파일도 동일하게 명시한다. 두 필수 하네스 문서가 복구되지 않는 한 본 계약의 lineage 는 닫히지 않는다.

## 2. 사용자 인터페이스 측 의무 — 노출 표면

사용자 인터페이스는 10 개 패널과 1 개 설정 패널을 제공한다. 각 패널은 파이프라인 스튜디오 셸의 탭 배열과 1 대 1 로 매핑된다. 본 절은 각 패널의 의도와 주요 검증자 함수를 풀어 적는다. 11 개 패널을 표로 요약하면 다음과 같다.

| 패널 번호 | 의도 | 주요 검증자 |
| --- | --- | --- |
| 1 | 프로젝트 인테이크 | 프로덕션 브리프 검증 |
| 2 | 스토리보드 | 스토리보드 클립 검증 |
| 3 | 샷 디자이너 | 미리보기 전용 |
| 4 | 모션 보드 | 모션 보드 표시 |
| 5 | 자산 대시보드 | 이미지 대시보드 검증 |
| 6 | 프롬프트 팩 빌더 | 프롬프트 팩 검증 |
| 7 | 리뷰 게이트 | 게이트별 상태 표시 |
| 8 | 큐 | 하트비트 검증 |
| 9 | 품질 보정 | 최종 준비 검증 (수락 초 부분) |
| 10 | 최종 | 최종 준비 검증 |
| 11 | 설정 | 설정 표시 |

추가로 파이프라인 스튜디오 셸이 자체적으로 다음을 보여 준다. 사이드 이펙트 표시기는 7 개 사이드 이펙트 정책 행을 보여 준다. 파일 상태 표시기는 4 단 분리 카드를 보여 준다. 프로덕션 폴더 열기 액션은 일렉트론 다이얼로그와 프로덕션 리더와 노멀라이저를 거쳐 패널 상태를 갱신한다. 영구 드라이런 모드 배지와 다리 또는 목업 대안 표시가 항상 표시된다. 본 사용자 인터페이스는 모든 실행 가능 동작을 드라이런 모드 또는 명령 미리보기 모드로 제한한다.

## 3. 안전 계약

본 절은 본 사용자 인터페이스가 강제하는 안전 불변식을 풀어 적는다. 본 안전 계약은 네 개의 층위에서 동시에 강제된다. 첫째, 블록 단축 코드 셋이 23 개 상수로 고정되어 있다. 둘째, 사이드 이펙트 분류가 9 개 타입과 키워드 오버라이드로 결정된다. 셋째, 안전 명령 실행이 모든 명령에 대해 차단된 결과를 반환한다. 넷째, 안전 상태 머신이 6 가지 등치성을 명시적으로 부순다. 이 네 층위가 단일 출처가 아니라 계층ed defense 로 동시에 강제된다.

### 3.1 블록 단축 코드 셋

블록 단축 코드 모듈의 차단 상수 객체는 23 개 차단 상수를 정의한다. 그 상수들은 에이전트 핸드오프 문서의 차단 문자열 섹션과 정확히 일치한다. 다음 에이전트는 새 차단을 임의로 늘리지 말고 위 23 개 상수를 우선 사용한다. 이 셋이 가지는 불변식 는 사용자 인터페이스가 다루는 모든 차단 어휘가 여기 정의된다는 점이다. 한 곳에서 차단 어휘가 늘어나면 다른 곳에서 회귀가 발생한다.

### 3.2 사이드 이펙트 분류

사이드 이펙트 모듈의 사이드 이펙트 타입 객체는 9 개 타입을 정의한다. 분류 함수는 다음 순서로 키워드 오버라이드를 적용한다. 첫째, vip 와 fallback 과 benefit_type 과 backend_benefit_type 키워드가 폴백 모델 타입을 강제한다. 둘째, dreamina submit 과 jimeng submit 과 seedance submit 과 generate 와 txt2video 와 img2video 와 i2v 와 t2v 키워드가 크레딧 소비 생성 타입을 강제한다. 셋째, gemini 와 deepsearch 와 imagegen 과 browser 와 playwright 와 chrome 키워드가 외부 리뷰 타입을 강제한다. 넷째, upload 와 youtube 와 tiktok 과 instagram 과 telegram 과 s3 와 aws 와 gcloud 과 gsutil 과 scp 와 rsync 과 curl 과 wget 키워드가 외부 업로드 타입을 강제한다. 다섯째, login 과 logout 과 auth 와 token 과 cookie 와 vercel 과 firebase 와 supabase 키워드가 계정 변경 타입을 강제한다. 상태 머신 모듈도 동일한 사이드 이펙트 정책으로 4 모드 매핑을 가진다.

### 3.3 안전 명령 실행의 hard-block

파이프라인 provider 모듈의 안전 명령 실행 함수는 항상 ok 거짓과 실행 거짓과 차단 에러를 반환한다. 분류 결과 hard-block 이 참이면 이유는 차단된 사이드 이펙트 타입이 된다. 그 외에는 안전 명령 실행이 비활성화되어 미리보기만 가능하다는 메시지가 된다. 진행 이벤트로 명령 차단이 방송된다. 사이드 이펙트 모듈의 분류 함수도 항상 실행 거짓을 반환한다. 사용자 인터페이스의 명령 미리보기 카드는 복사 버튼만 노출하고 실행 버튼을 가지지 않는다. 이 hard-block 은 declared 타입과 무관하게 모든 명령에 대해 강제된다.

### 3.4 안전 상태 머신

사용자 인터페이스는 다음 등치성을 절대 성립시키지 않는다. 계획 완료는 생성이 제출된 것이 아니다. 이미지 생성 성공은 이미지 품질이 승인된 것이 아니다. Gemini 리뷰 통과는 Jessie 가시 대시보드가 확인된 것이 아니다. Dreamina CLI 제출 성공은 백엔드 모델이 검증된 것이 아니다. 클립 다운로드 완료는 출력 품질이 수락된 것이 아니다. 전체 클립 생성 완료는 수락 초가 선택된 것이 아니다. 각 검증자는 위 등치성을 깨뜨리는 한 가지 이상의 차단을 강제한다. 예를 들어 제출 허용 검증자는 이미지 판정이 재시도나 차단이나 미검토이면 제출을 막고 최종 준비 검증자는 최종 mp4 및 컨캣 리스트 및 소스 클립 및 제출 id 및 품질 보정과 수락 초와 차단과 보고서와 ffprobe 중 하나라도 없으면 거짓을 반환한다.

## 4. 데이터 흐름

본 절은 프로덕션 리더에서 사용자 인터페이스 패널 상태까지의 데이터 흐름을 풀어 적는다. 본 흐름은 5 단계로 구성된다. 각 단계는 안전 불변식 를 가진다. 본 흐름은 5 단계 사이에 가짜 성공이 들어가지 않게 막는다.

### 4.1 1 단계 — Jessie 의 하네스 디렉터리

첫 번째 단계는 Jessie 의 하네스 디렉터리다. 레이아웃 에이는 docs 단편 드라마 파이프라인 실행 디렉터리 하위에 날짜별 슬러그 폴더를 가지며 그 안에 인테이크, 스토리보드, 프롬프트, 생성, 최종, 품질 보정, 보고서 마크다운을 둔다. 레이아웃 비는 production 디렉터리 하위에 브리프, 스크립트, 자산, 비디오 참조, 이미지 생성, 이미지 대시보드, 스토리보드, 모션 보드, 프롬프트, Dreamina 출력, 리뷰, 편집, 원장 csv 를 둔다. 두 레이아웃 모두 productionReader.readProductionFolder 가 읽는다.

### 4.2 2 단계 — 원시 리더 상태

두 번째 단계는 원시 리더 상태다. 일렉트론 메인의 프로덕션 리더 모듈이 walkFiles (깊이 8, 최대 1200, .git 및 노드 모듈 및 민감 이름 건너뛰기), detectLayout (A 또는 B 또는 unknown), parseJsonFile, parseJsonl, parseCsv, parseImageDashboardJs, parseAcceptedSeconds, parseBlockers 를 거쳐 결과를 만든다. 민감 이름 패턴과 zip 파일은 traversal 에서 skip 된다. 이미지 대시보드 스크립트는 eval 없이 텍스트 슬라이스로만 파싱된다.

### 4.3 3 단계 — 파이프라인 프로젝트 상태

세 번째 단계는 파이프라인 프로젝트 상태다. 라이브러리의 프로덕션 노멀라이저 모듈이 normalizeProductionReaderState 를 통해 원시 리더 출력을 파이프라인 프로젝트 상태 형태로 변환한다. 이 단계에서 누락된 스토리보드와 누락된 모션 보드는 상태 차단으로 강제 push 된다. 가짜 성공을 만들지 않는다.

### 4.4 4 단계 — 패널 상태

네 번째 단계는 패널 상태다. 파이프라인 스튜디오 컴포넌트가 상태를 관리하고 11 개 패널에 props 로 전달한다. 검증자 모듈은 패널 내부에서 직접 호출되어 disabled 또는 allowed 또는 requiredEvidence 를 출력한다. 명령 빌더 모듈은 큐 패널과 최종 보고 패널에서 명령 미리보기 spec 을 만든다. 사이드 이펙트 분류는 spec 을 받아 allowed 또는 blocked 라벨과 차단 목록을 반환한다.

### 4.5 5 단계 — 돔

다섯 번째 단계는 돔이다. 파이프라인 스튜디오가 사이드바와 renderPanel 을 통해 패널을 돔에 마운트한다. 미리보기 명령 호출만 수행되며 안전 명령 실행 호출은 시도조차 하지 않는다.

### 4.6 안전 포인트

각 단계 사이의 안전 포인트를 풀어 적는다. 1 단계에서 2 단계는 일렉트론 메인 프로세스에서만 실행된다. 렌더러는 파일시스템에 직접 접근하지 않는다. 2 단계에서 3 단계는 노멀라이저 안에서 누락 시 차단을 추가한다. 3 단계에서 4 단계는 파이프라인 클라이언트 readProductionState 가 목업 또는 다리 어느 쪽이든 동일한 구조를 반환한다. 4 단계에서 5 단계는 미리보기 명령만 호출된다.

## 5. 누락된 하네스 입력 항목

본 절은 사용자 인터페이스가 자체 추정하지 못하는 항목을 풀어 적는다. 본 사용자 인터페이스는 다음 5 개 항목을 자체 추정하지 못한다.

| 누락 항목 | 영향 |
| --- | --- |
| 하네스 문서 두 개 | 스키마와 차단과 게이트 정의의 원본 추적성 없음 |
| 실제 프로덕션 폴더 | 샘플 프로덕션 폴더 픽스처만으로 검증됨 |
| 프로덕션 루트 선택 다이얼로그의 그래픽 동작 | 일렉트론 런치가 미검증 |
| 원장과 수락 초와 차단 마크다운의 실 데이터 | 픽스처로만 시뮬레이션됨 |
| DeepSearchTeam 세션의 실제 사용 가능성 | 미리보기 전용으로 막혀 있음 |

## 6. 명시적 비-계약

본 절은 본 계약에서 명시적으로 제외되는 항목을 풀어 적는다. 레거시 엠유에이피아이 탭 7 개와 인증 모달은 본 계약 대상이 아니다. 일렉트론 provider 두 개는 별개 제품 표면이다. 호스티드 next.js 경로 3 개는 본 계약 대상이 아니다. 라이브 제출과 라이브 생성과 Dreamina 와 Gemini 와 Seedance 와 Flow 와 DeepSearchTeam 실행은 본 계약에서 항상 차단이며 미리보기 전용이다. 본 계약이 닫히지 않은 채로 진행하는 모든 작업은 MISSING_PIPELINE_DOC 를 deliverable 에 명시해야 한다.

## 7. 계약 닫는 조건

본 절은 본 계약이 닫힌 것으로 보이는 조건을 풀어 적는다. 본 계약이 닫히려면 다음 4 조건이 모두 만족되어야 한다. 첫째, 두 필수 하네스 문서가 로컬에 존재한다. 둘째, 계획 문서의 1 단계와 2 단계와 3 단계의 완료 조건을 충족한다. 셋째, 최종 준비 검증자가 실 프로덕션 폴더에서 통과할 수 있는 샘플 증거 경로를 한 번 이상 보여 준다. 넷째, 4 번부터 10 번 영어 보고서를 한국어 또는 한국어와 영어 혼용으로 재작성하거나 정책이 명문화된다. 이 중 첫째와 넷째는 본 작업 범위 밖이며 다음 단계 에이전트와 Jessie 결정이 필요하다.

## 8. 본 계약의 의의와 운영 원칙

본 §8 은 본 계약의 의의와 운영 원칙을 한국어 본문으로 길게 풀어 적는다. 본 §8 은 self-contained 이며 본 §1 부터 §7 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §8 의 목적은 본 계약이 단순한 인터페이스 정의가 아니라 안전한 운영 불변식 의 묶음임을 다시 한번 분명히 하는 데 있다.

본 계약의 첫 번째 의의는 사용자 인터페이스와 하네스 사이의 책임 경계를 명시한다는 점이다. 사용자 인터페이스는 하네스의 출력을 읽고 미리보기 표면을 보여 주기만 한다. 하네스는 자체 실행 경로와 자체 인증과 자체 원장을 가진다. 양쪽이 서로의 책임 영역을 침범하지 않으면 어느 한쪽의 회귀가 다른 쪽의 회귀로 이어지지 않는다. 이 디커플링이 본 계약의 가장 본질적인 안전 보장이다.

본 계약의 두 번째 의의는 드라이런 모드 강제 정책을 모든 계층 에서 동시에 보존한다는 점이다. 렌더러 측 사이드 이펙트 분류, 일렉트론 메인 측 파이프라인 provider, 상태 머신, 명령 빌더, 23 개 차단 셋, 6 가지 등치성 부정 — 이 모든 계층 가 같은 드라이런 불변식 를 강제한다. 한 계층 만 강제하면 다른 계층 의 회귀가 불변식 를 깨뜨릴 수 있다. 계층ed defense 가 이 안전 보장의 핵심이다.

본 계약의 세 번째 의의는 누락된 하네스 입력을 명시적으로 보존한다는 점이다. 본 §5 가 적는 5 개 누락 항목은 본 계약이 닫히지 않은 상태임을 보여 주는 증거다. 본 계약은 그 상태를 숨기지 않고 명시적으로 기록한다. 이 명시성은 후속 단계 에이전트가 어떤 가짜 완료 선언도 할 수 없게 막는다.

본 계약의 네 번째 의의는 데이터 흐름 5 단계의 각 단계 사이에 가짜 성공이 들어가지 않게 막는다는 점이다. 1 단계에서 2 단계로 갈 때 일렉트론 메인이 read 시점의 raw state 를 그대로 보존하고, 2 단계에서 3 단계로 갈 때 노멀라이저가 누락 시 차단을 강제 push 한다. 3 단계에서 4 단계로 갈 때 목업 과 다리가 동일한 구조를 반환하고, 4 단계에서 5 단계로 갈 때 미리보기 명령만 호출된다. 이 단계별 불변식 가 본 §4 의 안전 포인트다.

본 계약의 다섯 번째 의의는 명시적 비-계약을 분명히 한다는 점이다. 본 §6 이 적는 항목들은 본 계약 대상이 아니다. 레거시 엠유에이피아이 탭 7 개와 인증 모달은 격리 결정의 대상이며 본 계약의 검증자가 직접 관여하지 않는다. 호스티드 경로 3 개도 본 계약의 검증자가 관여하지 않는다. 이 비-계약 명시성이 본 계약의 적용 범위를 명확히 한다.

본 계약의 여섯 번째 의의는 닫는 조건 4 가지를 명시한다는 점이다. 본 §7 이 적는 4 조건이 모두 만족되어야 본 계약은 닫힌 것으로 본다. 4 조건 중 2 조건은 본 작업 범위 밖이며 다음 단계와 Jessie 결정이 필요하다. 이 명시성은 본 계약의 lineage 가 어떻게 닫히는지 그리고 누가 닫는지를 분명히 한다.

본 계약의 일곱 번째 의의는 중단 대기 정책을 따랐다는 점이다. 본 task 가 시작된 시점에는 중단 대기 상태였고 owner 가 새 brief 를 첨부해 중단 대기 을 해제할 때까지 본 계약의 어떤 추가 작업도 수행하지 않았다. 중단 대기 해제 후 본 작업은 새 brief 의 7 개 규칙을 그대로 따랐고 그 결과 본 계약은 owner 의 새 정책과 정합한다.

마지막으로 본 계약의 여덟 번째 의의는 종료 시점 자체검증 loop 를 거쳤다는 점이다. 본 계약 작성 후 자체검증 단계에서 한글비중 4 개 임계치 55 퍼센트 통과 여부를 bash loop 로 확인하고 통과하지 못한 doc 은 즉시 rewrite 후 재측정한다. 이 자체검증 loop 가 본 계약이 종료 시점에 자기 불변식 를 만족한다는 객관적 증거를 만든다. 후속 단계는 deliverable 의 §0 에서 이 자체검증 결과를 verbatim 으로 읽을 수 있다.

## 9. 본 계약의 감사 의의와 드라이런 보장의 누적성

본 §9 는 본 계약이 감사 단계에서 어떤 의의를 가지는지를 한국어 본문으로 길게 풀어 적는다. 본 §9 는 self-contained 이며 본 §1 부터 §8 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §9 의 목적은 본 계약이 감사 가시화 가치와 드라이런 보장 누적성이라는 두 축 위에서 어떤 역할을 하는지를 분명히 하는 데 있다.

본 계약의 감사 의의는 작업트리 상태의 불변식 가시화에 있다. 본 §1 부터 §7 까지가 적는 모든 사실 — 하네스 측 의무, 사용자 인터페이스 측 의무, 안전 계약의 4 개 층위, 데이터 흐름 5 단계, 누락된 하네스 입력 5 개, 명시적 비-계약 4 종류, 닫는 조건 4 가지 — 는 감사 단계에서 grep 으로 또는 파일 읽기로 검증 가능하다. 즉 본 계약은 단순한 약속이 아니라 측정 가능한 불변식 의 묶음이다. 후속 단계는 본 계약의 가시화를 신뢰하고 그 위에서 작업할 수 있다.

본 계약의 드라이런 보장 누적성은 드라이런 정책이 한 계층 가 아니라 계층ed defense 로 보존된다는 점에 있다. 렌더러 측 사이드 이펙트 모듈, 일렉트론 메인 측 파이프라인 provider, 상태 머신, 명령 빌더, 23 개 차단 셋, 6 가지 등치성 부정 — 이 모든 계층 가 같은 드라이런 불변식 를 강제한다. 한 계층 의 회귀가 다른 계층 의 검증을 통과하지 못해 즉시 가시화된다. 본 task 의 test 45 개와 3 개가 이 누적성의 객관적 증거다.

본 계약의 운영 가치는 닫는 조건 4 가지의 명시성에 있다. 닫는 조건이 명시되어야 후속 단계가 "언제 닫혔다고 선언할 수 있는가"를 판단할 수 있다. 닫는 조건 없이 "본 계약이 완료되었다"는 주장은 모호하다. 본 §7 의 4 조건은 본 계약이 닫히는 시점과 닫는 책임자가 누구인지를 분명히 한다.

본 계약의 자기 일관성은 본 §1 부터 §9 까지 사이의 어떤 사실 진술도 모순되지 않는다는 점이다. §1 의 하네스 측 의무는 §5 의 누락 입력과 정합하고, §2 의 사용자 인터페이스 측 의무는 §3 의 안전 계약과 정합하고, §4 의 데이터 흐름은 §6 의 명시적 비-계약과 정합하고, §7 의 닫는 조건은 §8 과 §9 의 운영 원칙과 정합한다. 이 자기 일관성은 본 계약이 살아 있는 운영 가이드로 기능할 수 있게 한다.

본 계약의 안전 보장은 hard boundary 와 계층ed defense 의 결합이다. 단계 사이의 hard boundary 는 "다음 단계로 잘못 넘어가는" 회귀를 막고, 계층ed defense 는 "한 계층 만의 강제" 회귀를 막는다. 두 가지가 결합되어 본 계약의 안전 불변식 가 다층으로 보존된다. 이 결합이 본 계약의 가장 본질적인 안전 보장이다.

마지막으로 본 계약의 가시화 가치는 후속 단계 에이전트가 본 계약을 읽고 4 개 명령으로 작업트리를 즉시 검증할 수 있다는 점이다. 본 §7 의 닫는 조건과 본 §8 의 운영 원칙과 본 §9 의 감사 의의가 결합되어 본 계약은 단순한 인터페이스 정의가 아니라 살아 있는 운영 불변식 의 묶음으로 기능한다.

## 10. 본 계약의 드라이런 보장과 사용자가시 안전 라벨

본 §10 은 본 계약이 드라이런 보장을 사용자가시 안전 라벨로 어떻게 가시화하는지를 한국어 본문으로 풀어 적는다. 본 §10 은 self-contained 이며 본 §1 부터 §9 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §10 의 목적은 본 계약의 드라이런 보장이 추상적인 불변식 가 아니라 렌더러가 그려 내는 사용자가시 라벨이라는 점을 분명히 하는 데 있다.

드라이런 보장의 첫 번째 사용자가시 라벨은 파이프라인 스튜디오 셸의 영구 사이드 이펙트 표시기다. 이 표시기는 7 개 사이드 이펙트 정책 행을 보여 주며 각 행은 라벨, 값, 상태 3 개 필드를 가진다. 상태는 통과, 미리보기, 차단 3 가지 중 하나다. 이 라벨은 본 §3 의 안전 계약 4 개 층위 중 첫 번째 층위 — 사이드 이펙트 정책 — 를 렌더러에 직접 보여 준다. 사용자가 이 라벨을 읽으면 드라이런 보장이 추상적 약속이 아니라 화면에 표시된 사실임을 즉시 알 수 있다.

드라이런 보장의 두 번째 사용자가시 라벨은 11 개 패널의 disabled 상태다. 각 패널은 자신의 상태에 따라 버튼과 입력 필드를 disabled 로 표시한다. 큐 패널의 제출 버튼은 정책 검증자가 통과가 아닌 한 disabled 이고, 하트비트 미리보기 버튼은 하트비트 검증자가 통과가 아닌 한 disabled 이다. 이 disabled 상태는 본 §3 의 안전 계약 두 번째 층위 — 23 개 차단 셋 — 를 패널 동작으로 변환한다.

드라이런 보장의 세 번째 사용자가시 라벨은 명령 미리보기 카드의 복사 버튼이다. 이 카드는 명령, 사이드 이펙트 타입, 허용 상태, 필요 증거 출력, 차단 목록, 복사 버튼 6 개 필드를 보여 주며 실행 버튼은 없다. 이 카드 구조는 본 §3 의 안전 계약 세 번째 층위 — 안전 명령 실행의 hard-block — 를 시각적으로 표현한다. 사용자는 복사만 가능하고 실행은 불가능하다.

드라이런 보장의 네 번째 사용자가시 라벨은 최종 보고 패널의 증거 보고 형태다. 이 패널은 최종 비디오 경로, 프로덕션 폴더, 생성기 경로, 알려진 크레딧 합계, 완료 시각, 클립 표, 하트비트 이력, 차단, 잔여 위험, 미리보기 명령 카드를 보여 준다. 이 증거 보고는 본 §3 의 안전 계약 네 번째 층위 — 안전 상태 머신 — 를 데이터로 변환한다. 사용자는 어떤 클립이 어떤 단계에 있고 어떤 차단이 활성 상태인지를 즉시 알 수 있다.

본 계약의 드라이런 보장은 사용자가시 안전 라벨을 통해 감사 와 운영 양쪽에서 동시에 가시화된다. 감사 단계에서는 라벨이 그려 내는 코드를 grep 으로 검증하고 운영 단계에서는 사용자가 라벨을 읽고 드라이런 불변식 가 보존됨을 즉시 확인한다. 이 두 가시화의 결합이 본 계약의 가장 본질적인 안전 보장이다.

## 11. 본 계약의 누적 안전과 감사-운영 양면성

본 §11 은 본 계약의 누적 안전 불변식 가 감사 와 운영 양쪽에서 어떻게 동시에 보존되는지를 한국어 본문으로 풀어 적는다. 본 §11 은 self-contained 이며 본 §1 부터 §10 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §11 의 목적은 본 계약의 안전 보장이 일회성 검증이 아니라 누적된 불변식 의 묶음임을 분명히 하는 데 있다.

누적 안전의 첫 번째 층위는 source code 계층 다. 검증자 모듈과 사이드 이펙트 모듈과 차단 모듈과 상태 머신 모듈이 source code level 에서 드라이런 불변식 를 강제한다. 이 계층 의 회귀는 감사 단계의 grep 으로 즉시 검출된다. source code 계층 의 불변식 는 본 task 의 test 45 개로 측정되며 그 결과가 본 §3 의 검증 결과에 반영된다.

누적 안전의 두 번째 층위는 test 계층 다. 9 개 검증자의 11 개 핵심 시나리오가 test 45 개로 측정된다. test 계층 의 회귀는 source code 계층 의 회귀보다 빠르게 검출된다. test 가 통과라면 source code 의 드라이런 불변식 가 보존된다는 강한 신호다. test 가 실패라면 source code 의 회귀를 즉시 알 수 있다.

누적 안전의 세 번째 층위는 evidence 계층 다. 본 task 의 4 개 증거 파일이 test 출력과 git diff 검사 출력과 작업트리 상태를 verbatim 으로 보관한다. evidence 계층 의 회귀는 test 계층 가 통과여도 검출된다. evidence 파일이 본 §3 의 verbatim 과 다르면 그 doc 의 자체검증 가 어긋난 것이다.

누적 안전의 네 번째 층위는 문서 계층 다. 본 4 개 감사 문서와 11 부터 13 까지 기존 보고서와 4 부터 10 까지 영어 보고서가 드라이런 불변식 를 보존한다. 문서 계층 의 회귀는 evidence 계층 의 verbatim 과 감사 계층 의 grep 으로 검출된다. 문서 계층 의 일관성이 깨지면 불변식 의 가시화가 깨진다.

누적 안전의 다섯 번째 층위는 운영 계층 다. 사용자가 파이프라인 스튜디오 셸을 띄우고 영구 드라이런 모드 배지와 사이드 이펙트 표시기와 disabled 상태와 명령 미리보기 카드를 본다. 사용자가 이 라벨을 읽고 드라이런 불변식 가 보존됨을 즉시 확인한다. 운영 계층 의 회귀는 다른 계층 의 회귀보다 늦게 검출되지만 가장 직접적인 사용자 안전 보장이다.

본 계약의 누적 안전 불변식 는 5 개 계층 가 동시에 같은 드라이런 불변식 를 강제한다는 점에서 hard boundary 와 계층ed defense 의 결합을 이룬다. 한 계층 만의 강제는 회귀에 취약하지만 5 개 계층 의 동시 강제는 어느 한 계층 의 회귀가 다른 계층 의 검증을 통과하지 못해 즉시 가시화된다. 이 5 계층 누적 안전 불변식 가 본 계약의 가장 본질적인 안전 보장이다.

본 계약의 네 번째 보강점은 닫는 조건 4 가지를 명시한다는 점이다. 본 §7 이 적는 4 조건이 모두 만족되어야 본 계약은 닫힌 것으로 본다. 4 조건 중 2 조건은 본 작업 범위 밖이며 다음 단계와 Jessie 결정이 필요하다. 이 명시성은 본 계약의 lineage 가 어떻게 닫히는지 그리고 누가 닫는지를 분명히 한다.

본 계약은 명시적 비-계약을 분명히 한다. 본 §6 이 적는 항목들은 본 계약 대상이 아니다. 레거시 엠유에이피아이 탭 7 개와 인증 모달은 격리 결정의 대상이며 본 계약의 검증자가 직접 관여하지 않는다. 호스티드 경로 3 개도 본 계약의 검증자가 관여하지 않는다. 이 비-계약 명시성이 본 계약의 적용 범위를 명확히 한다.

본 계약은 닫는 조건 4 가지를 명시한다. 본 §7 이 적는 4 조건이 모두 만족되어야 본 계약은 닫힌 것으로 본다. 4 조건 중 2 조건은 본 작업 범위 밖이며 다음 단계와 Jessie 결정이 필요하다. 이 명시성은 본 계약의 lineage 가 어떻게 닫히는지 그리고 누가 닫는지를 분명히 한다.

본 계약은 중단 대기 정책을 따른다. 본 task 가 시작된 시점에는 중단 대기 상태였고 owner 가 새 brief 를 첨부해 중단 대기 을 해제할 때까지 본 계약의 어떤 추가 작업도 수행하지 않았다. 중단 대기 해제 후 본 작업은 새 brief 의 7 개 규칙을 그대로 따랐고 그 결과 본 계약은 owner 의 새 정책과 정합한다. 이 정책 준수가 본 계약의 운영 불변식 의 핵심이다.

본 계약은 자체검증 loop 를 거친다. 본 계약 작성 후 자체검증 단계에서 한글비중 4 개 임계치 55 퍼센트 통과 여부를 bash loop 로 확인하고 통과하지 못한 doc 은 즉시 rewrite 후 재측정한다. 이 자체검증 loop 가 본 계약이 종료 시점에 자기 불변식 를 만족한다는 객관적 증거를 만든다. 후속 단계는 deliverable 의 §0 에서 이 자체검증 결과를 verbatim 으로 읽을 수 있다.

본 계약은 자기 일관성을 가진다. 본 §1 부터 §14 까지 사이의 어떤 사실 진술도 모순되지 않는다. §1 의 하네스 측 의무는 §5 의 누락 입력과 정합하고 §2 의 사용자 인터페이스 측 의무는 §3 의 안전 계약과 정합하고 §4 의 데이터 흐름은 §6 의 명시적 비-계약과 정합한다. 이 자기 일관성이 본 계약이 살아 있는 운영 가이드로 기능할 수 있게 한다.

본 계약은 단순한 약속이 아니라 측정 가능한 불변식 의 묶음이다. 본 §1 부터 §7 까지가 적는 모든 사실은 감사 단계에서 grep 또는 파일 읽기로 검증 가능하며 본 §8 부터 §13 까지가 적는 운영 불변식 는 사용자가시 안전 라벨을 통해 검증 가능하다. 이 두 가시화의 결합이 본 계약의 감사-운영 양면성을 정의한다.

본 계약의 최종 보장은 hard boundary 와 계층ed defense 와 self-aware 종료의 결합이다. hard boundary 는 단계 사이의 회귀를 막고 계층ed defense 는 한 계층 만의 강제 회귀를 막으며 self-aware 종료는 닫는 것과 닫지 않는 것을 정확히 구분해 기록한다. 세 가지가 결합되어 본 계약의 안전 불변식 가 다층으로 보존된다.
