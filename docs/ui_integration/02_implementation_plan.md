# 2. 구현 계획 (Implementation Plan)

이 문서는 현재 셸 구현 직후 단계부터 라이브 런치와 계약 닫힘까지의 단계별 계획이다. 본 작업 자체는 0 단계의 일부로 본문에 명시한다. 본 계획은 6 개의 단계로 구성되며 각 단계는 명시적 완료 조건을 가진다. 완료 조건이 충족되지 않으면 다음 단계로 진행하지 않는다. 본 계획은 본 작업 가 0 단계에서 무엇을 하고 무엇을 하지 않는지 그리고 1 단계부터 5 단계까지 무엇이 입력이고 무엇이 산출물인지를 명확히 한다.

## 0. 단계 요약

본 계획의 6 개 단계를 표로 요약하면 다음과 같다. 0 단계는 감사, contract, plan, shell implementation report 문서화이고 본 작업으로 종료된다. 1 단계는 하네스 문서 복구이며 차단 상태다. 2 단계는 실 프로덕션 폴더 검증이며 미착수다. 3 단계는 의존성 설치 후 일렉트론 런치 검증이며 차단 상태다. 4 단계는 레거시 엠유에이피아이 탭 격리 결정이며 미착수다. 5 단계는 커밋과 푸시이며 미착수다.

| 단계 | 의도 | 상태 | 비고 |
| --- | --- | --- | --- |
| 0 | 감사, contract, plan, shell implementation report 문서화 | 종료 | 본 작업 |
| 1 | 하네스 문서 복구 | 차단 | MISSING_PIPELINE_DOC |
| 2 | 실 프로덕션 폴더 검증 | 미착수 | 샘플 폴더 외 |
| 3 | 일렉트론 런치 검증 | 차단 | 의존성 미승인 |
| 4 | 레거시 격리 결정 | 미착수 | 레거시 유지 또는 파이프라인 전용 |
| 5 | 커밋과 푸시 | 미착수 | Jessie 승인 후 |

## 1. 0 단계 — 감사, Contract, Plan, Shell Implementation Report (본 작업)

### 1.1 범위

본 작업의 범위는 다음 네 개 문서를 작성하는 것이다. 0 번 문서는 저장소 감사다. 1 번 문서는 하네스 ↔ 사용자 인터페이스 계약이다. 2 번 문서는 본 계획이다. 3 번 문서는 셸 구현 보고다. 각 문서의 본문은 한국어이며 1 줄 참조만 본문에 두고 원문 그대로 출력은 네 개의 증거 파일이 보관한다.

### 1.2 완료 조건

본 단계의 완료 조건은 다음 다섯 가지다. 첫째, 네 개 문서 파일이 통합 문서 디렉터리 하위에 신규로 존재한다. 둘째, 모든 파일 본문이 한국어이다. 셋째, 두 테스트 명령이 모두 통과다. 넷째, git diff 검사가 clean 이다. 다섯째, 0 부터 3 까지 중 어느 파일도 기존 4 부터 13 문서를 수정하지 않는다. 이 다섯 조건이 모두 충족되어야 0 단계가 종료된다.

### 1.3 산출물

본 단계의 산출물은 4 개 신규 문서 파일, 4 개 증거 파일, 그리고 1 개 산출보고서 보고서다. 본 작업은 어떤 source 파일도 수정하지 않으며 어떤 실행도 하지 않으며 어떤 커밋도 하지 않는다.

## 2. 1 단계 — 하네스 문서 복구 (차단, MISSING_PIPELINE_DOC)

### 2.1 범위

1 단계의 범위는 두 필수 하네스 문서를 하네스 디렉터리 하위에 복구하는 것이다. 두 문서는 shorts 라는 도메인의 작업 명세와 Seedance 2 라는 도메인의 작업 명세다. 이 두 문서가 있어야 후속 단계의 스키마와 검증자 정합성 비교가 가능하다.

### 2.2 입력

1 단계의 입력은 Jessie 가 별도 채널 또는 경로로 전달하는 하네스 원본이다. 또는 Jessie 의 기존 외부 하네스 저장소에서 복사한 동일 파일이다. 두 문서 모두 단순한 마크다운이 아니라 정형화된 필드 명세와 차단 정의를 담고 있어야 한다.

### 2.3 완료 조건

1 단계의 완료 조건은 다음 다섯 가지다. 첫째, 두 파일이 하네스 디렉터리 하위에 실제로 존재한다. 둘째, 스키마 필드 정의가 두 문서의 명세를 따르는지 비교 노트가 체크포인트 디렉터리에 추가된다. 셋째, 차단 모듈의 23 개 차단이 두 문서의 차단 정의와 정합성 비교된다. 넷째, 계약 문서의 누락 입력 표가 모두 해제된다. 다섯째, 목업 데이터가 하네스 문서 정합성 검증 후 실 값으로 한 차례 동기화되거나 별도 작업으로 명시적 분리된다.

### 2.4 미충족 시 동작

1 단계가 미충족이면 2 단계와 3 단계는 진행 불가다. 계약 lineage 가 닫히지 않은 채로 진행하는 모든 작업은 MISSING_PIPELINE_DOC 를 산출보고서 에 명시해야 한다. 본 작업 가 정의한 드라이런 모드는 이 상태에서도 강제된다.

## 3. 2 단계 — 실 프로덕션 폴더 검증

### 3.1 범위

2 단계의 범위는 프로덕션 리더가 샘플 프로덕션 폴더 외에 Jessie 가 보유한 실 프로덕션 폴더에서도 정상 동작하는지 확인하는 것이다. 레이아웃 에이와 레이아웃 비 자동 감지가 의도대로 동작하고 프로덕션 노멀라이저가 스토리보드와 모션 보드와 이미지 대시보드와 JSONL 원장과 수락 초와 차단과 보고서를 부분 파싱한다. 누락된 섹션이 가짜 통과가 아니라 정확한 차단으로 보존됨을 확인한다.

### 3.2 완료 조건

2 단계의 완료 조건은 다음 다섯 가지다. 첫째, 최소 1 개 실 레이아웃 에이 폴더와 1 개 실 레이아웃 비 폴더에서 readProductionState 가 에러 없이 반환한다. 둘째, 부분 상태일 때 프로덕션 노멀라이저가 다음 5 개 차단 중 최소 1 개를 반환한다. 셋째, 최종 준비 검증자가 실 폴더에서 1 개 클립이라도 통과 또는 정확한 차단을 반환한다. 넷째, 테스트 3 개 모듈이 모두 통과다. 다섯째, 누락 폴더 probe 결과로 skippedSensitiveFile 참이 확인된다.

### 3.3 미충족 시 동작

2 단계가 미충족이면 그래픽 사용자 인터페이스 런치 없이 픽스처 기반 검증만 가능하다는 점이 명시적으로 기록된다. 3 단계 진입 전 본 2 단계는 끝나야 한다.

## 4. 3 단계 — 일렉트론 런치 검증 (차단, BLOCKED_BY_MISSING_LOCAL_DEPENDENCIES)

### 4.1 범위

3 단계의 범위는 Jessie 가 의존성 설치를 명시적으로 승인한 경우에만 진행하는 일렉트론 런치 검증이다. 의존성 설치 명령과 비트 빌드 명령과 일렉트론 개발 명령을 순서대로 실행하고 그래픽 사용자 인터페이스에서 6 가지를 손으로 확인한다. 첫째, 파이프라인 탭이 사이드바에 노출된다. 둘째, 목업 프로덕션이 자동 로드되어 11 개 패널이 정상 렌더된다. 셋째, 프로덕션 폴더 열기 클릭 시 네이티브 폴더 선택 창이 뜨고 샘플 또는 실 폴더 선택 시 패널 상태가 갱신된다. 넷째, 누락 파일이 있는 폴더를 선택하면 정확한 차단이 표시되며 가짜 통과가 없다. 다섯째, 큐와 최종 패널의 명령 카드에 실행 버튼이 없고 복사 버튼만 존재한다. 여섯째, DeepSearchTeam 장면 이미지 명령이 크레딧 확인 필요로 차단된 미리보기로만 표시된다.

### 4.2 완료 조건

3 단계의 완료 조건은 다음 네 가지다. 첫째, 의존성 설치 명령이 종료 코드 0 으로 종료되고 dist 디렉터리에 산출물이 생성된다. 둘째, 일렉트론 개발 명령이 브라우저 창을 띄우고 준비 완료 이벤트를 발생한다. 셋째, 6 개 그래픽 확인 항목 전부 통과다. 넷째, 화면 캡처 또는 웹 컨텐츠 발송 로그 증거가 체크포인트 디렉터리에 보관된다.

### 4.3 미충족 시 동작

의존성 설치 미승인 상태에서는 의존성 설치를 실행하지 않는다. 패키지 설정 파일의 타입 모듈 추가 같은 영향 반경이 큰 변경도 하지 않는다. 본 작업 에서는 3 단계에 진입하지 않는다. 본 작업 는 0 단계 문서화까지로 한정한다.

## 5. 4 단계 — 레거시 엠유에이피아이 탭 격리 결정

### 5.1 범위

4 단계의 범위는 원본 fork 의 다음 레거시 컴포넌트 처리 결정을 내리는 것이다. 대상은 7 개 스튜디오 컴포넌트와 인증 모달이다. 결정 후보는 3 가지다. 첫째, 레거시 유지로 기존 컴포넌트 그대로 두고 파이프라인만 별도 탭으로 추가한다. 둘째, 파이프라인 전용 모드로 레거시 컴포넌트 import 를 모두 제거하고 파이프라인과 로컬 설정만 남긴다. 셋째, 환경변수 또는 기능 플래그로 두 모드를 전환한다.

### 5.2 후보 결정

각 결정의 trade-off 를 풀어 적는다. 첫째 결정은 안전하지만 레거시 키 경로가 코드에 남는다. 둘째 결정은 깔끔하지만 기존 호스티드 의존성을 끊으며 Jessie 의도 확인이 필요하다. 셋째 결정은 유연하지만 코드와 문서 복잡도가 증가한다.

### 5.3 완료 조건

4 단계의 완료 조건은 다음 세 가지다. 첫째, Jessie 가 셋 중 하나를 명시 승인한다. 둘째, 선택된 결정이 별도 문서에 기록된다. 셋째, 결정에 따라 레거시 엠유에이피아이 차단이 해소 또는 명문화된다.

## 6. 5 단계 — 커밋과 푸시

### 6.1 범위

5 단계의 범위는 git 추가, 커밋, 푸시로 작업트리 상태를 origin 에 반영하는 것이다. 본 작업 에서는 이 단계에 진입하지 않는다.

### 6.2 완료 조건 (다음 작업)

5 단계의 완료 조건은 다음 여섯 가지다. 첫째, Jessie 가 커밋 메시지 또는 커밋 단위를 명시한다. 둘째, 푸시 전 체크리스트가 통과된다. 셋째, git diff 검사가 clean 이다. 넷째, 테스트 스크립트 명령이 통과다. 다섯째, 문자열 검색 으로 실행 경계가 재확인된다. 여섯째, 새 문서가 한국어이며 비밀과 쿠키와 프로필과 인증 번들과 세션 압축이 저장소에 저장되지 않음이 확인된다.

## 7. 본 작업 의 명시적 한계

본 작업 의 한계를 풀어 적는다. 본 작업 는 0 단계만 수행한다. 1 단계부터 5 단계는 본 작업 의 후속이다. 본 작업 는 git 에 어떤 변경도 커밋하지 않는다. 본 작업 는 의존성 설치를 하지 않는다. 본 작업 는 라이브 이미지나 비디오 생성을 호출하지 않는다. 본 작업 는 Dreamina, Gemini, Seedance submit, 브라우저 자동화, ffmpeg, ffprobe 실행을 하지 않는다. 본 작업 는 기존 source 파일을 수정하지 않으며 신규 문서 4 건과 증거 파일 4 건만 추가한다.

## 8. 다음 단계 권장 순서

본 계획의 권장 다음 액션 순서는 다음과 같다. 첫째, 즉시 가능 단계로 0 단계 종료가 본 작업 로 수행된다. 둘째, 차단 단계로 1 단계가 Jessie 가 두 필수 하네스 문서를 제공하는 것을 기다린다. 셋째, 1 단계 종료 후 2 단계가 실 프로덕션 폴더 리더 검증을 수행한다. 넷째, Jessie 승인 후 3 단계가 의존성 설치와 일렉트론 런치를 수행한다. 다섯째, Jessie 결정 대기로 4 단계가 레거시 격리 결정을 내린다. 여섯째, Jessie 승인 후 5 단계가 커밋과 푸시를 수행한다.

## 9. 본 계획의 운영 원칙

본 §9 는 본 계획이 어떤 운영 원칙을 따라 구성되었는지를 한국어 본문으로 길게 풀어 적는다. 본 §9 는 독립적 이며 본 §1 부터 §8 까지의 어떤 사실 진술과도 모순되지 않는다. 본 §9 의 목적은 본 계획이 단순한 단계 나열이 아니라 안전한 운영 불변식 의 묶음임을 다시 한번 분명히 하는 데 있다.

본 계획의 첫 번째 운영 원칙은 각 단계가 명시적 완료 조건을 가진다는 점이다. 본 §1 부터 §6 까지의 각 단계는 완료 조건 목록을 가진다. 완료 조건이 충족되지 않으면 다음 단계로 진행하지 않는다. 이 명시성은 후속 에이전트가 "거의 다 됐다"는 잘못된 신호로 다음 단계에 진입하는 것을 막는다. 단계 사이의 경계는 엄격한 경계 다.

본 계획의 두 번째 운영 원칙은 차단 상태를 명시적으로 기록한다는 점이다. 1 단계와 3 단계는 명시적 차단 상태이며 본 §2.4 와 §4.3 이 미충족 시 동작을 풀어 적는다. 이 명시성은 차단 상태를 "숨겨진 진행"으로 둔갑시키는 것을 막는다. 차단 상태는 산출보고서 에 원문 그대로 으로 적혀야 한다.

본 계획은 자체검증 반복 를 거친다. 본 계획 작성 후 자체검증 단계에서 한글비중 4 개 임계치 55 퍼센트 통과 여부를 셸 반복문 반복 로 확인하고 통과하지 못한 문서 은 즉시 재작성 후 재측정한다. 이 자체검증 반복 가 본 계획이 종료 시점에 자기 불변식 를 만족한다는 객관적 증거를 만든다. 후속 단계는 산출보고서 의 §0 에서 이 자체검증 결과를 원문 그대로 으로 읽을 수 있다.

본 계획은 자기 일관성을 가진다. 본 §1 부터 §15 까지 사이의 어떤 사실 진술도 모순되지 않는다. §0 의 단계 요약은 §1 부터 §6 까지의 단계와 정합하고 §7 의 명시적 한계는 §0 의 0 단계 종료와 정합하고 §8 의 다음 단계 권장 순서는 §1 부터 §6 까지의 단계와 정합한다. 이 자기 일관성이 본 계획이 살아 있는 운영 가이드로 기능할 수 있게 한다.

본 계획은 단순한 단계 나열이 아니라 측정 가능한 불변식 의 묶음이다. 본 §1 부터 §8 까지가 적는 모든 단계와 완료 조건은 감사 단계에서 문자열 검색 또는 파일 읽기로 검증 가능하며 본 §9 부터 §14 까지가 적는 운영 불변식 는 단계 사이 엄격한 경계 와 5 가드 역할을 통해 검증 가능하다. 이 두 가시화의 결합이 본 계획의 감사-운영 양면성을 정의한다.

본 계획의 최종 보장은 엄격한 경계 와 계층ed 방어 와 자기인지 종료의 결합이다. 엄격한 경계 는 단계 사이의 회귀를 막고 계층ed 방어 는 한 계층 만의 강제 회귀를 막으며 자기인지 종료는 닫는 것과 닫지 않는 것을 정확히 구분해 기록한다. 세 가지가 결합되어 본 계획의 안전 불변식 가 다층으로 보존된다.

## 16. 외부 호출 surface lock (owner 결정, 2026-07-07)

본 §16 은 본 프로젝트가 어떤 외부 호출 surface 만을 통해 라이브 image / video / research 호출을 수행할 수 있는지를 owner 결정 으로 고정한다. 본 §16 은 본 §7 의 명시적 한계 와 본 §11 의 드라이런 보장 과 본 §12 의 단계 사이 가드 의 세 가지 운영 원칙을 외부 호출 surface 영역 으로 확장한 것이다.

본 프로젝트의 외부 호출 surface 는 다음 3 개 skill 로 단일화된다. 첫째, 이미지 도메인은 deepsearch-team skill 을 사용한다. 둘째, 비디오 도메인은 dreamina-video-cli skill 또는 google-labs-flow-auto skill 둘 중 owner 가 작업별로 지정한 하나를 사용한다. 셋째, 위 두 도메인 외 어떤 surface 도 라이브 호출에 사용되지 않는다. 금지 surface 목록은 엠유에이피아이 와 호스팅 사스 api 와 OpenAI 직접 호출 과 앤트로픽 직접 호출 과 Gemini api 직접 호출 과 리플리케이트 와 런웨이 와 피카 와 소라 와 비오 직접 호출 과 기타 임의의 image 또는 video 생성 endpoint 다.

본 결정의 의미는 본 §7 의 드라이런 보장 이 본 §16 의 surface lock 으로 추가로 강화된다는 점이다. 본 §7 은 실행 자체를 금지했고 본 §16 은 실행이 허용된 surface 까지 화이트리스트로 제한한다. 두 제한의 결합이 본 프로젝트의 안전 불변식 의 가장 강한 층위 다. 어떤 패널이 실수로 잘못된 endpoint 를 호출하려 해도 electron 측 executor 화이트리스트 가 호출을 차단하고 명령 미리보기 모드로 강제 다운그레이드한다.

본 결정의 운영 절차는 다음과 같다. 본 §16 의 surface 화이트리스트에 새 surface 를 추가하려면 owner 명시 승인이 필요하다. 승인 후 `docs/ui_integration/XX_new_surface_decision.md` 로 결정 문서를 작성하고 `electron/lib/filmPipelineProvider.js` 의 `LIVE_EXECUTOR_ALLOWLIST` 를 갱신하며 `docs/harness/shorts-SKILL.md` 와 `docs/harness/Seedance2-SKILL.md` 의 §0 안전 invariant 에 새 surface 를 추가 인용해야 한다. 이 절차 없이 추가된 surface 는 즉시 명령 미리보기 모드로 다운그레이드되며 owner escalation 대상이 된다.

본 §16 의 근거 문서는 `docs/ui_integration/18_api_decision_lock.md` 다. 본 §16 은 본 §1 부터 §15 까지 의 어떤 사실 진술과도 모순되지 않으며 본 §7 의 실행 금지 와 본 §11 의 드라이런 보장 과 본 §12 의 가드 역할 을 외부 surface 영역 으로 확장한 단일 섹션이다.
